## 2024-05-29 - Staggered Concurrent Requests to bypass HTTP 429 limits
**Learning:** Sequential `await` calls spaced with arbitrary delays (like `setTimeout(..., 1000)`) lead to a linear increase in response time, creating a severe bottleneck when many API calls are needed. Using purely concurrent `Promise.all` can easily trigger HTTP 429 rate limit errors when making multiple requests to an external API like `api-open.data.gov.sg`.
**Action:** Use a "staggered concurrency" approach to combine the speed of `Promise.all` with rate-limit compliance. Wrap the promises in a delayed thunk (e.g., using `setTimeout`) inside the `Promise.all` mapping, incrementing the delay by a safe margin (e.g., 200ms) for each successive request. This significantly reduces total wait time while reliably preventing HTTP 429 errors.
